Git： 代码合并

早在之前，就已经了解过git中几种不同合并代码方法的异同。相信这是在一个较大的项目中都会遇到的抉择。本文的主要目的其实是为了推荐一个git的交互式学习网站，同时记下几个比较有用的命令。

首先是正题，是的，就是这么直白。

学习git的交互式网站
Learn Git Branching

HEAD~/^

HEAD默认指向当前分支的最近一次提交，而~/^则是对其指向位置的修改。当然也可以在HEAD后直接跟commit的id，将HEAD直接指向需要的commit。

  • ~是对当前分支纵向偏移的位置，即始终在这个分支上，不会指向其他的分支。~与~1效果相同，~n则是移向之前的第n个commit。
  • ^是对HEAD之前的commit横向偏移的位置。如果只是^，则就是将HEAD移向之前的commit，^1与其结果相同。而如果当前commit是由两个commitmerge到一起的话，则HEAD将会指向原本不在这个分支的commit的位置。如果^之后的数字大于之前横向的commit数，则会报错。

在之前推荐的交互系统中，你可以使用git checkout HEAD~/^来观察HEAD的变化。例如：

  • 使HEAD指向C4
    git checkout HEAD~
git checkout HEAD~1
git checkout HEAD^
git checkout HEAD^1
  • 使HEAD指向C2
    git checkout HEAD^2
  • 使HEAD指向C1
    git checkout HEAD~2
git checkout HEAD^~
git checkout HEAD^2~

git rebase master bugFix则会把C3和C4按顺序放到C2之后，HEAD指向C4，在bugFix分支上。如下图。

  • merge产生的C5的提交信息为Merge branch "bugFix" into branch "master"。而C3和C4的提交则按提交顺序插入到历史中。即后来的人只知道在这个地方有过merge操作，但不知道这个操作里面的具体步骤等，即增加了开发人员理解项目的难度。
  • rebase最大的好处是你的项目历史会非常整洁。首先，它不像git merge 那样引入不必要的合并提交。其次，如上图所示，rebase导致最后的项目历史呈现出完美的线性——你可以从项目终点到起点浏览而不需要任何的Fork。这让你更容易使用查看历史。但是如果你错误的使用这个命令，则会重写项目的历史，影响在公共分支上的其他人。

  • 一种更好的合并方式是git rebase -i， 它允许你更改并入新分支的提交，即对分支上提交历史完整的控制。一般来说，这被用于将feature分支并入master分支之前，清理混乱的历史。

git commit --amend

这个命令其实是对当前分支最后一次的提交信息进行修改。即如果你在本地提交之后，发现提交信息有些错误，这时候你就可以使用这个命令对其进行修改。

git cherry-pick


将处于其他分支上的C3,C4,C7合并到master分支上。则git cherry-pick master C3 C4 C7
